아래는 **2D 어드벤처 게임을 실제로 하나 완성한다는 기준**으로 정리한 개발 절차입니다. 예시는 **탑다운 2D 어드벤처** 기준으로 설명하겠습니다.
예: 플레이어가 마을을 돌아다니며 NPC와 대화하고, 아이템을 얻고, 퍼즐을 풀고, 다음 지역을 여는 게임.

---

# 0단계. 먼저 게임의 범위를 아주 작게 정한다

2D 어드벤처 게임은 욕심을 내면 금방 커집니다. 처음부터 “젤다 같은 게임”, “언더테일 같은 게임”, “스타듀밸리 같은 게임”을 목표로 잡으면 거의 실패합니다.

처음 목표는 이렇게 잡는 것이 좋습니다.

> **플레이 시간 10~20분짜리 작은 어드벤처 게임 1개 완성**

최소 완성 버전의 예시는 이 정도입니다.

| 항목   | 최소 목표             |
| ---- | ----------------- |
| 맵    | 마을 1개, 숲 1개       |
| 플레이어 | 걷기, 상호작용, 아이템 줍기  |
| NPC  | 3명                |
| 대화   | NPC별 대화 2~3개      |
| 아이템  | 5개                |
| 퀘스트  | 2개                |
| 퍼즐   | 1~2개              |
| 저장   | 자동 저장 또는 수동 저장 1개 |
| 엔딩   | 짧은 엔딩 1개          |

가장 중요한 원칙은 이것입니다.

> **처음부터 큰 게임을 만들지 말고, “작지만 끝까지 완성된 게임”을 먼저 만든다.**

---

# 1단계. 게임 콘셉트를 한 문장으로 정한다

먼저 게임을 한 문장으로 설명할 수 있어야 합니다.

나쁜 예:

> 귀엽고 감동적인 모험 게임

좋은 예:

> 기억을 잃은 아이가 작은 섬을 탐험하며 주민들의 부탁을 해결하고, 사라진 음악 상자를 찾는 2D 탑다운 어드벤처 게임

더 구체적인 예:

> 플레이어는 마을과 숲을 오가며 NPC와 대화하고, 아이템을 모아 잠긴 문을 열고, 마지막에 잃어버린 물건을 찾아주는 짧은 2D 어드벤처 게임

이 한 문장이 이후 모든 결정을 좌우합니다.

# 2단계. 핵심 재미를 정한다

2D 어드벤처 게임의 재미는 보통 아래 요소 중 일부에서 나옵니다.

| 재미 요소 | 설명                   |
| ----- | -------------------- |
| 탐험    | 숨겨진 장소, 새로운 맵 발견     |
| 대화    | NPC와 이야기하며 세계관 이해    |
| 수집    | 열쇠, 재료, 단서, 장비 획득    |
| 퍼즐    | 문 열기, 순서 맞추기, 스위치 조작 |
| 성장    | 능력 해금, 이동 범위 확장      |
| 스토리   | 사건 해결, 감정적 결말        |
| 분위기   | 음악, 색감, 연출, 사운드      |

처음 게임에서는 모든 것을 넣지 말고 **3개만 고르는 것**이 좋습니다.

예를 들면:

> 탐험 + 대화 + 아이템 퍼즐

또는:

> 스토리 + NPC 대화 + 간단한 퀘스트

음악이론 모바일 게임 웹앱과 연결한다면 이런 방향도 가능합니다.

> 탐험 + 음악 퍼즐 + NPC 퀘스트
> 예: 마을 주민들이 잃어버린 음표를 찾아주고, 코드 진행 퍼즐을 풀어 다음 지역을 여는 게임

# 3단계. 개발 도구를 선택한다

2D 어드벤처 게임은 여러 도구로 만들 수 있습니다.

| 도구        | 적합한 경우                          |
| --------- | ------------------------------- |
| Godot     | 인디 2D 게임, 무료 오픈소스, 가벼운 프로젝트     |
| Unity     | 모바일, PC, 콘솔까지 확장 가능하고 자료가 많은 경우 |
| GameMaker | 순수 2D 게임을 빠르게 만들고 싶은 경우         |
| Phaser    | 웹 브라우저에서 실행되는 2D 게임을 만들 경우      |
| PixiJS    | 웹에서 직접 2D 렌더링과 인터랙션을 세밀하게 만들 경우 |

Godot은 공식적으로 2D와 3D 게임을 만들 수 있는 무료 오픈소스 게임 엔진으로 소개되어 있고, Godot 공식 문서에는 첫 2D 게임을 만드는 튜토리얼도 제공됩니다. ([Godot Engine][1]) Unity는 공식 문서에서 스프라이트, 타일맵, 2D 물리 시스템 등을 포함한 2D 개발 기능을 안내하고 있습니다. ([Unity 문서][2]) GameMaker는 공식 사이트에서 2D 게임 제작용 개발 도구로 소개되며, Windows, macOS, Linux, Android, iOS, HTML5 등 다양한 플랫폼 출력을 지원한다고 설명합니다. ([GameMaker][3])

웹앱으로 만들 계획이라면 **Phaser, PixiJS, Canvas API** 쪽이 잘 맞습니다. Phaser는 HTML5 게임 프레임워크로 WebGL과 Canvas 렌더링을 지원한다고 공식 저장소에서 설명하고, MDN은 Canvas API가 JavaScript와 `<canvas>` 요소로 애니메이션과 게임 그래픽을 그리는 데 사용될 수 있다고 안내합니다. ([GitHub][4])

처음 만드는 사람에게 추천하는 선택은 이렇습니다.

| 목표          | 추천                              |
| ----------- | ------------------------------- |
| 웹에서 바로 실행   | Phaser + TypeScript             |
| 모바일 앱까지 생각  | Godot 또는 Unity                  |
| 빠른 2D 프로토타입 | Godot 또는 GameMaker              |
| 음악이론 웹게임    | Phaser 또는 React + Canvas/PixiJS |

# 4단계. 게임 디자인 문서, GDD를 만든다

GDD는 Game Design Document의 약자입니다. 거창하게 만들 필요는 없습니다. 처음에는 3~5페이지면 충분합니다.

## GDD에 반드시 들어갈 내용

### 1. 게임 기본 정보

```text
게임 제목: 잃어버린 음표의 숲
장르: 2D 탑다운 어드벤처
플랫폼: 모바일 웹
플레이 시간: 15분
대상 유저: 음악 기초를 배우는 초보자
핵심 재미: 탐험, 대화, 음악 퍼즐
```

### 2. 핵심 플레이 흐름

```text
1. 플레이어가 마을에서 시작한다.
2. NPC와 대화해서 문제를 듣는다.
3. 숲으로 이동한다.
4. 아이템 또는 단서를 찾는다.
5. 간단한 퍼즐을 푼다.
6. NPC에게 돌아간다.
7. 보상을 받고 다음 지역이 열린다.
8. 마지막 퍼즐을 풀면 엔딩이 나온다.
```

### 3. 조작 방식

PC 기준:

```text
이동: WASD 또는 방향키
상호작용: Space 또는 E
메뉴: Esc
```

모바일 기준:

```text
이동: 가상 조이스틱 또는 터치 이동
상호작용: 화면 버튼
메뉴: 우측 상단 버튼
```

### 4. 주요 시스템

```text
플레이어 이동
카메라 추적
맵 충돌
대화 시스템
아이템 시스템
인벤토리
퀘스트 시스템
퍼즐 시스템
저장/불러오기
사운드
UI
```

### 5. 엔딩 조건

```text
최종 아이템 “황금 음표”를 얻고 마을 중앙의 음악 상자에 사용하면 엔딩.
```

# 5단계. 핵심 게임 루프를 만든다

게임 루프는 플레이어가 계속 반복하는 행동 구조입니다.

2D 어드벤처 게임의 기본 루프는 보통 이렇습니다.

```text
탐험한다
→ NPC 또는 오브젝트와 상호작용한다
→ 단서나 퀘스트를 얻는다
→ 아이템을 찾는다
→ 퍼즐을 푼다
→ 새로운 장소가 열린다
→ 스토리가 진행된다
```

이 루프가 재미있어야 게임 전체가 재미있습니다.

예를 들어 음악이론 어드벤처라면:

```text
마을 탐험
→ NPC가 “도 레 미” 순서를 잊었다고 말함
→ 숲에서 음표 조각을 찾음
→ 올바른 음계 순서로 배치
→ 문이 열림
→ 다음 지역으로 이동
```

# 8단계. 가장 먼저 “플레이어 이동”을 만든다

첫 번째 목표는 캐릭터가 화면에서 움직이는 것입니다.

구현 순서는 이렇습니다.

```text
1. 빈 맵을 만든다.
2. 플레이어 스프라이트를 배치한다.
3. 키보드 입력을 받는다.
4. 입력 방향에 따라 위치를 바꾼다.
5. 이동 속도를 조절한다.
6. 방향별 애니메이션을 붙인다.
7. 벽과 충돌하게 만든다.
8. 카메라가 플레이어를 따라오게 만든다.
```

이 단계에서 확인할 것:

```text
상하좌우 이동이 자연스러운가?
대각선 이동 속도가 너무 빠르지 않은가?
벽을 뚫고 지나가지 않는가?
카메라가 너무 떨리지 않는가?
모바일에서 조작이 불편하지 않은가?
```

초반에는 캐릭터 그래픽이 없어도 됩니다. 네모 박스로 시작해도 괜찮습니다.

# 9단계. 맵과 충돌을 만든다

2D 어드벤처 게임의 맵은 보통 타일맵으로 만듭니다.

맵에 필요한 레이어는 다음과 같습니다.

```text
Ground Layer: 바닥
Decoration Layer: 풀, 꽃, 돌 같은 장식
Collision Layer: 벽, 나무, 물, 절벽
Object Layer: 문, 상자, NPC, 아이템
Above Layer: 플레이어보다 위에 그려질 지붕, 나뭇잎
Trigger Layer: 이벤트 발생 영역
```

맵 하나를 만들 때는 이런 체크리스트를 사용합니다.

```text
시작 위치가 명확한가?
플레이어가 어디로 가야 하는지 자연스럽게 보이는가?
막힌 길과 갈 수 있는 길이 구분되는가?
NPC와 아이템이 눈에 잘 띄는가?
숨겨진 요소가 너무 불공평하지 않은가?
맵 크기가 너무 크지 않은가?
```

초기 맵은 작게 만드는 것이 좋습니다.

```text
마을: 30 x 20 타일
숲: 40 x 25 타일
동굴: 25 x 20 타일
```

처음부터 100 x 100 타일짜리 맵을 만들면 테스트와 수정이 힘들어집니다.

# 10단계. 상호작용 시스템을 만든다

어드벤처 게임에서 가장 중요한 기능 중 하나는 **상호작용**입니다.

플레이어가 NPC, 문, 상자, 표지판, 아이템과 상호작용해야 합니다.

상호작용 시스템은 이렇게 설계합니다.

```text
플레이어가 오브젝트 근처에 간다.
→ 상호작용 가능 표시가 뜬다.
→ 버튼을 누른다.
→ 오브젝트 타입에 맞는 이벤트가 실행된다.
```

오브젝트 타입 예시:

```text
NPC: 대화 시작
Door: 문 열기 또는 잠김 메시지 출력
Chest: 아이템 획득
Sign: 설명 문구 출력
Puzzle: 퍼즐 UI 열기
Portal: 다른 맵으로 이동
```

오브젝트 데이터는 이런 식으로 관리할 수 있습니다.

```json
{
  "id": "village_old_man",
  "type": "npc",
  "name": "촌장",
  "dialogueId": "old_man_intro",
  "questId": "find_lost_note"
}
```

이렇게 데이터로 관리하면 나중에 NPC가 많아져도 코드를 계속 수정하지 않아도 됩니다.

---

# 11단계. 대화 시스템을 만든다

2D 어드벤처 게임의 핵심은 대화입니다.

처음에는 아주 단순한 구조로 시작하면 됩니다.

## 기본 대화 시스템 기능

```text
대화창 열기
텍스트 한 글자씩 출력
다음 문장 넘기기
대화 종료
NPC 이름 표시
선택지 표시
조건에 따라 다른 대화 출력
```

대화 데이터 예시:

```json
{
  "id": "old_man_intro",
  "speaker": "촌장",
  "lines": [
    "마을의 음악 상자가 멈춰버렸단다.",
    "숲 어딘가에 잃어버린 음표가 있을 거야.",
    "찾아와 준다면 길을 열어주마."
  ],
  "nextQuest": "find_lost_note"
}
```

선택지가 있는 대화는 이렇게 설계할 수 있습니다.

```json
{
  "id": "npc_question",
  "speaker": "음악 선생님",
  "line": "C장조의 으뜸음은 무엇일까요?",
  "choices": [
    {
      "text": "도",
      "result": "correct"
    },
    {
      "text": "파",
      "result": "wrong"
    }
  ]
}
```

대화 시스템에서 중요한 것은 코드보다 **데이터 구조**입니다.
대화를 코드 안에 직접 쓰기 시작하면 나중에 수정하기가 매우 힘들어집니다.

---

# 12단계. 아이템과 인벤토리를 만든다

어드벤처 게임에서는 아이템이 진행의 열쇠가 됩니다.

아이템 종류는 처음에는 단순하게 나눕니다.

```text
Key Item: 진행용 아이템
Consumable: 사용하면 사라지는 아이템
Collectible: 수집용 아이템
Quest Item: 퀘스트용 아이템
```

처음 게임에서는 대부분 **Key Item**과 **Quest Item**만 있어도 충분합니다.

아이템 데이터 예시:

```json
{
  "id": "lost_note_c",
  "name": "도 음표",
  "type": "quest",
  "description": "음악 상자를 고치는 데 필요한 첫 번째 음표.",
  "icon": "note_c.png"
}
```

인벤토리 기능은 처음에 이렇게만 만들면 됩니다.

```text
아이템 추가
아이템 보유 여부 확인
아이템 사용
아이템 제거
아이템 목록 UI 표시
```

문을 여는 예시는 다음과 같습니다.

```text
플레이어가 잠긴 문과 상호작용한다.
→ “숲의 열쇠”를 가지고 있는지 확인한다.
→ 있으면 문을 연다.
→ 없으면 “열쇠가 필요하다” 메시지를 출력한다.
```

# 13단계. 퀘스트 시스템을 만든다

퀘스트 시스템은 복잡하게 만들 필요 없습니다.

처음에는 아래 상태만 있어도 충분합니다.

```text
not_started: 시작 전
active: 진행 중
completed: 완료
rewarded: 보상 수령
```

퀘스트 데이터 예시:

```json
{
  "id": "find_lost_note",
  "title": "잃어버린 음표",
  "description": "숲에서 도 음표를 찾아 촌장에게 가져가자.",
  "startNpc": "village_old_man",
  "requiredItems": ["lost_note_c"],
  "reward": {
    "unlockMap": "forest_deep"
  }
}
```

퀘스트 진행 흐름은 이렇게 설계합니다.

```text
NPC 대화
→ 퀘스트 시작
→ 목표 아이템 획득
→ NPC에게 돌아감
→ 아이템 확인
→ 퀘스트 완료
→ 보상 지급
→ 새 지역 해금
```

퀘스트를 만들 때 주의할 점:

```text
목표가 명확해야 한다.
플레이어가 어디로 가야 하는지 힌트가 있어야 한다.
보상이 있어야 한다.
완료 여부가 UI나 대화로 확인되어야 한다.
```

# 14단계. 퍼즐 시스템을 만든다

2D 어드벤처에서 퍼즐은 게임 진행을 재미있게 만드는 장치입니다.

처음에는 아래 정도의 퍼즐이 적당합니다.

```text
스위치 3개를 올바른 순서로 누르기
아이템을 제단에 놓기
문양 순서 맞추기
음표 순서 맞추기
NPC가 준 힌트를 보고 정답 입력하기
```

음악이론 게임이라면 이런 퍼즐이 좋습니다.

```text
도-레-미-파 순서대로 음표 배치
C-E-G를 골라 C major 코드 만들기
높은음자리표에서 특정 음 찾기
리듬 패턴을 듣고 같은 순서로 터치하기
장조/단조 분위기 구분하기
```

퍼즐 설계 공식은 다음과 같습니다.

```text
힌트 제공
→ 플레이어가 관찰
→ 조작
→ 피드백
→ 성공 또는 실패
→ 성공 시 보상
```

나쁜 퍼즐:

```text
힌트 없이 아무 순서로나 스위치를 눌러야 함
```

좋은 퍼즐:

```text
표지판에 “낮은 소리에서 높은 소리로 길이 열린다”라고 쓰여 있음
→ 플레이어가 도, 미, 솔 순서로 발판을 밟음
→ 문이 열림
```

# 15단계. 게임 상태 관리 구조를 만든다

게임에는 여러 상태가 있습니다.

```text
탐험 중
대화 중
인벤토리 열림
퍼즐 UI 열림
컷신 재생 중
맵 이동 중
일시정지
게임 오버
엔딩
```

상태 관리를 제대로 하지 않으면 이런 문제가 생깁니다.

```text
대화 중인데 캐릭터가 움직임
퍼즐 중인데 인벤토리가 열림
맵 이동 중에 버튼을 여러 번 눌러 버그 발생
컷신 중에 NPC와 대화가 시작됨
```

따라서 전역 게임 상태를 둡니다.

```text
GameState = Explore
GameState = Dialogue
GameState = Inventory
GameState = Puzzle
GameState = Cutscene
GameState = Pause
```

상태별 허용 행동은 이렇게 정합니다.

| 상태        | 이동 | 상호작용 |       메뉴 | 대화 넘기기 |
| --------- | -: | ---: | -------: | -----: |
| Explore   | 가능 |   가능 |       가능 |     불가 |
| Dialogue  | 불가 |   불가 | 불가 또는 제한 |     가능 |
| Inventory | 불가 |   불가 |       가능 |     불가 |
| Puzzle    | 불가 |   불가 | 불가 또는 제한 |     불가 |
| Cutscene  | 불가 |   불가 |       불가 |     자동 |

이 구조를 초반에 잡아두면 버그가 크게 줄어듭니다.

# 16단계. 저장과 불러오기를 만든다

저장 기능은 후반에 넣으면 꼬일 수 있습니다.
기본 구조는 중반부터 준비하는 것이 좋습니다.

저장해야 할 정보는 다음과 같습니다.

```text
현재 맵 ID
플레이어 위치
보유 아이템 목록
완료한 퀘스트 목록
진행 중인 퀘스트 목록
열린 문 목록
획득한 상자 목록
퍼즐 해결 여부
옵션 설정
플레이 시간
```

저장 데이터 예시:

```json
{
  "currentMap": "village",
  "playerPosition": {
    "x": 120,
    "y": 240
  },
  "inventory": ["lost_note_c", "forest_key"],
  "quests": {
    "find_lost_note": "completed"
  },
  "flags": {
    "forest_gate_open": true,
    "chest_001_opened": true
  }
}
```

여기서 중요한 개념은 **flag**입니다.

flag는 게임 진행 여부를 기록하는 스위치입니다.

예:

```text
촌장과 첫 대화를 했는가?
숲의 문이 열렸는가?
상자 1번을 열었는가?
첫 번째 퍼즐을 풀었는가?
엔딩을 봤는가?
```

어드벤처 게임은 결국 flag 관리가 핵심입니다.

# 17단계. UI를 만든다

2D 어드벤처 게임에 필요한 UI는 다음과 같습니다.

```text
대화창
상호작용 안내 표시
인벤토리 창
아이템 설명창
퀘스트 목록
일시정지 메뉴
설정 메뉴
저장 완료 표시
모바일 조작 버튼
```

처음에는 예쁘게 만들기보다 **읽기 쉽고 사용하기 쉽게** 만드는 것이 중요합니다.

대화창 체크리스트:

```text
글자가 충분히 큰가?
모바일에서도 읽을 수 있는가?
NPC 이름이 보이는가?
다음 버튼이 명확한가?
대화가 너무 빨리 넘어가지 않는가?
배경과 글자의 대비가 충분한가?
```

모바일 UI 체크리스트:

```text
버튼이 손가락으로 누르기 충분히 큰가?
중요한 UI가 화면 가장자리에 너무 붙어 있지 않은가?
가로 모드와 세로 모드 중 무엇을 기준으로 할지 정했는가?
작은 화면에서도 대화창이 잘 보이는가?
```

# 18단계. 아트 스타일을 정한다

2D 어드벤처 게임은 아트 스타일이 매우 중요합니다.
하지만 처음부터 완벽한 그래픽을 만들 필요는 없습니다.

개발 단계별 아트는 이렇게 진행합니다.

## 1차: 임시 그래픽

```text
플레이어 = 파란 네모
NPC = 초록 네모
벽 = 검은 네모
아이템 = 노란 원
```

## 2차: 기본 스프라이트

```text
플레이어 걷기 애니메이션
NPC 기본 이미지
타일셋
아이템 아이콘
UI 박스
```

## 3차: 완성 그래픽

```text
방향별 걷기 애니메이션
NPC 표정
맵 장식
조명 효과
컷신 이미지
엔딩 이미지
```

아트 스타일은 처음에 이런 식으로 문서화합니다.

```text
해상도: 16x16 또는 32x32 픽셀 타일
시점: 탑다운
색감: 따뜻한 파스텔톤
캐릭터 비율: 1.5등신
분위기: 아늑하고 약간 신비로운 느낌
```

주의할 점은 **아트 해상도를 섞지 않는 것**입니다.

예를 들어 16x16 픽셀 타일과 64x64 고해상도 캐릭터를 섞으면 어색해 보입니다.

# 19단계. 사운드와 음악을 넣는다

어드벤처 게임에서는 사운드가 분위기를 크게 좌우합니다.

필요한 사운드는 다음과 같습니다.

```text
걷기 효과음
상호작용 효과음
아이템 획득 효과음
문 열림 효과음
대화 넘김 효과음
퍼즐 성공 효과음
퍼즐 실패 효과음
맵별 배경음악
엔딩 음악
```

사운드는 너무 늦게 넣지 않는 것이 좋습니다.
임시 효과음이라도 넣으면 게임의 느낌을 훨씬 빨리 확인할 수 있습니다.

음악이론 게임이라면 사운드는 단순 장식이 아니라 게임 규칙의 일부가 될 수 있습니다.

예:

```text
정답 음을 누르면 맑은 소리
오답 음을 누르면 불협화음
C major 코드를 완성하면 코드가 실제로 재생됨
리듬 퍼즐에서 박자에 맞춰 터치해야 함
```

# 20단계. 첫 번째 프로토타입을 만든다

프로토타입의 목표는 “재미가 있는지 확인하는 것”입니다.
그래픽이 예쁠 필요는 없습니다.

## 1차 프로토타입 목표

```text
플레이어가 움직인다.
NPC와 대화한다.
아이템을 줍는다.
아이템으로 문을 연다.
다음 맵으로 이동한다.
간단한 엔딩이 나온다.
```

이 정도면 이미 어드벤처 게임의 뼈대가 있습니다.

1차 프로토타입은 이렇게 구성하면 좋습니다.

```text
맵 1개
NPC 1명
아이템 1개
문 1개
대화 3줄
엔딩 메시지 1개
```

예:

```text
촌장: “숲의 열쇠를 찾아오게.”
플레이어가 열쇠를 줍는다.
문에 열쇠를 사용한다.
문이 열린다.
“다음 지역으로 향했다.” 메시지 출력.
```

이 단계에서 확인해야 할 것은 딱 하나입니다.

# 21단계. 버티컬 슬라이스를 만든다

버티컬 슬라이스는 게임의 작은 완성본입니다.

프로토타입이 “작동 확인용”이라면, 버티컬 슬라이스는 “최종 게임의 품질을 작게 보여주는 버전”입니다.

버티컬 슬라이스에는 다음이 들어갑니다.

```text
완성된 느낌의 작은 맵 1개
최종 스타일에 가까운 플레이어 그래픽
NPC 1~2명
대화
아이템
퍼즐
효과음
배경음악
UI
저장
짧은 목표와 보상
```

즉, 게임 전체의 10%만 만들지만 품질은 최종에 가깝게 만듭니다.

이 단계에서 다음 질문에 답해야 합니다.

```text
이 게임을 더 크게 만들 가치가 있는가?
조작이 재미있는가?
대화가 지루하지 않은가?
퍼즐이 이해되는가?
아트 스타일이 잘 맞는가?
모바일에서 불편하지 않은가?
```

# 22단계. 전체 콘텐츠를 설계한다

버티컬 슬라이스가 괜찮다면 전체 콘텐츠를 설계합니다.

작은 2D 어드벤처의 전체 구조 예시는 다음과 같습니다.

```text
지역 1: 마을
- 튜토리얼
- NPC 3명
- 첫 퀘스트
- 기본 조작 학습

지역 2: 숲
- 첫 번째 퍼즐
- 아이템 수집
- 숨겨진 길

지역 3: 동굴
- 두 번째 퍼즐
- 마지막 단서
- 최종 문

지역 4: 음악 상자 방
- 최종 퍼즐
- 엔딩
```

콘텐츠 표를 만들어 관리하면 좋습니다.

| ID   | 콘텐츠        | 위치   | 조건            | 보상    |
| ---- | ---------- | ---- | ------------- | ----- |
| Q001 | 잃어버린 음표 찾기 | 마을/숲 | 촌장과 대화        | 숲길 개방 |
| P001 | 음계 순서 퍼즐   | 숲    | 음표 3개 획득      | 동굴 열쇠 |
| Q002 | 동굴 입구 열기   | 숲    | 동굴 열쇠 보유      | 동굴 입장 |
| P002 | 코드 맞추기     | 동굴   | C, E, G 단서 획득 | 황금 음표 |
| END  | 음악 상자 수리   | 마을   | 황금 음표 보유      | 엔딩    |

# 23단계. 맵 디자인을 구체화한다

맵은 예쁜 것보다 **플레이어가 자연스럽게 길을 찾을 수 있는 것**이 중요합니다.

좋은 맵은 다음 구조를 가집니다.

```text
시작 지점
→ 눈에 띄는 목표
→ 막힌 길
→ 우회 경로
→ 아이템 발견
→ 다시 돌아와 문 열기
→ 새 지역 개방
```

예:

```text
마을 중앙에서 시작
→ 오른쪽에 잠긴 숲 문이 보임
→ 촌장이 열쇠 이야기를 함
→ 왼쪽 집 뒤에서 열쇠 발견
→ 숲 문으로 돌아감
→ 숲 진입
```

맵을 만들 때는 플레이어 시선을 유도해야 합니다.

시선 유도 방법:

```text
길을 밝게 만든다.
중요한 오브젝트를 중앙에 둔다.
NPC 주변에 빈 공간을 둔다.
문이나 길목에 독특한 장식을 둔다.
막힌 길에는 명확한 이유를 보여준다.
```

나쁜 맵:

```text
모든 곳이 똑같이 생김
어디로 가야 할지 알 수 없음
중요한 아이템이 배경과 구분되지 않음
불필요하게 넓음
```

좋은 맵:

```text
목표가 보임
돌아올 이유가 있음
짧은 동선 안에 발견과 보상이 있음
```

# 24단계. NPC와 대사를 작성한다

NPC는 단순히 정보를 주는 존재가 아니라 게임 세계를 살아있게 만드는 요소입니다.

NPC마다 역할을 정합니다.

| NPC | 역할            |
| --- | ------------- |
| 촌장  | 메인 퀘스트 제공     |
| 아이  | 힌트 제공         |
| 상인  | 아이템 설명        |
| 음악가 | 퍼즐 규칙 설명      |
| 길지기 | 진행 차단 또는 문 개방 |

대사를 쓸 때는 짧고 명확하게 씁니다.

나쁜 대사:

```text
아주 오래전부터 이 마을에는 수많은 전설이 있었고 그 전설은 대대로 내려왔으며...
```

좋은 대사:

```text
숲의 문은 음표 3개가 있어야 열린단다.
첫 번째 음표는 오래된 우물 근처에서 봤어.
```

대화는 보통 다음 3가지 역할 중 하나를 해야 합니다.

```text
목표를 알려준다.
힌트를 준다.
캐릭터성을 보여준다.
```

아무 역할도 없는 대사는 줄이는 것이 좋습니다.

---

# 25단계. 컷신과 연출을 만든다

처음에는 컷신을 복잡하게 만들지 않아도 됩니다.

간단한 컷신 구조:

```text
플레이어 조작 잠금
카메라 이동
NPC 이동
대화 출력
효과음 재생
화면 페이드
게임 상태 변경
조작 해제
```

예:

```text
플레이어가 황금 음표를 음악 상자에 사용한다.
→ 화면이 어두워진다.
→ 음악 상자가 빛난다.
→ 짧은 멜로디가 재생된다.
→ 마을 사람들이 모인다.
→ 촌장이 감사 인사를 한다.
→ 엔딩 크레딧
```

컷신은 많이 넣기보다 중요한 순간에만 넣는 것이 좋습니다.

```text
게임 시작
새 지역 개방
중요한 아이템 획득
최종 퍼즐 성공
엔딩
```

# 26단계. 제작 일정을 세운다

처음 게임은 8주 정도로 잡으면 좋습니다.

## 8주 개발 일정 예시

| 주차  | 목표                       |
| --- | ------------------------ |
| 1주차 | 콘셉트, GDD, 도구 선택, 프로젝트 세팅 |
| 2주차 | 플레이어 이동, 카메라, 충돌, 기본 맵   |
| 3주차 | 상호작용, NPC, 대화 시스템        |
| 4주차 | 아이템, 인벤토리, 문 열기, 퀘스트     |
| 5주차 | 퍼즐 시스템, 저장/불러오기          |
| 6주차 | 맵 2~3개 제작, 전체 콘텐츠 연결     |
| 7주차 | 아트, 사운드, UI, 연출 개선       |
| 8주차 | 테스트, 버그 수정, 빌드, 배포       |

더 작은 일정으로는 4주 버전도 가능합니다.

## 4주 미니 프로젝트 일정

| 주차  | 목표              |
| --- | --------------- |
| 1주차 | 이동, 충돌, 맵, 상호작용 |
| 2주차 | 대화, 아이템, 퀘스트    |
| 3주차 | 퍼즐, UI, 사운드     |
| 4주차 | 완성, 테스트, 배포     |

# 27단계. 작업을 티켓 단위로 쪼갠다

“대화 시스템 만들기”는 너무 큰 작업입니다.
이렇게 쪼개야 합니다.

```text
대화창 UI 만들기
NPC 이름 표시하기
텍스트 출력하기
다음 문장 넘기기
대화 종료하기
플레이어 이동 잠그기
대화 데이터 JSON으로 불러오기
선택지 표시하기
퀘스트 상태에 따라 다른 대화 보여주기
```

작업 관리 예시:

```text
[필수] 플레이어 이동 구현
[필수] 카메라 추적 구현
[필수] 벽 충돌 구현
[필수] NPC 상호작용 구현
[필수] 대화창 구현
[필수] 아이템 획득 구현
[필수] 문 열기 구현
[선택] NPC 표정 변화
[선택] 대화 타이핑 효과
[선택] 화면 흔들림 효과
```

항상 필수 기능과 선택 기능을 분리해야 합니다.

# 28단계. 개발 순서는 반드시 “뼈대 → 콘텐츠 → polish”로 간다

초보자가 자주 하는 실수는 예쁜 화면부터 만드는 것입니다.

권장 순서는 다음과 같습니다.

```text
1. 회색 박스와 임시 이미지로 기능 구현
2. 전체 게임을 처음부터 끝까지 연결
3. 실제 그래픽으로 교체
4. 사운드 추가
5. 애니메이션 추가
6. 연출 개선
7. 최적화
8. 버그 수정
```

즉, 처음부터 예쁜 타이틀 화면을 만들 필요가 없습니다.
가장 먼저 해야 하는 일은 이것입니다.

> 시작 버튼을 누르면 게임이 시작되고, 플레이어가 움직이고, 목표를 달성하면 엔딩이 나오는 구조 만들기

# 29단계. 테스트를 계속한다

게임 개발에서 테스트는 마지막 단계가 아닙니다.
매일 해야 합니다.

## 매일 확인할 것

```text
게임이 실행되는가?
새로 만든 기능이 작동하는가?
기존 기능이 망가지지 않았는가?
저장이 깨지지 않는가?
맵 이동이 정상인가?
대화 중 이동이 막히는가?
아이템 중복 획득 버그가 없는가?
```

## 플레이테스트 질문

테스터에게 이렇게 물어보면 좋습니다.

```text
지금 무엇을 해야 하는지 알겠나요?
길을 잃은 부분이 있었나요?
대화가 너무 길었나요?
퍼즐의 힌트가 충분했나요?
조작이 불편했나요?
어떤 순간이 가장 재미있었나요?
그만하고 싶어진 순간이 있었나요?
```

중요한 점은 테스터에게 설명하지 말고 그냥 플레이하게 하는 것입니다.
설명을 해야만 이해한다면 게임 안의 안내가 부족한 것입니다.

# 30단계. 버그 목록을 관리한다

버그는 반드시 문서로 관리합니다.

버그 기록 예시:

| ID   | 버그            | 재현 방법             | 심각도 | 상태   |
| ---- | ------------- | ----------------- | --- | ---- |
| B001 | 대화 중 이동 가능    | NPC에게 말 걸고 방향키 입력 | 높음  | 수정 전 |
| B002 | 열쇠를 두 번 획득 가능 | 맵 이동 후 다시 상자 열기   | 중간  | 수정 중 |
| B003 | 모바일에서 버튼이 작음  | iPhone 화면에서 테스트   | 중간  | 수정 전 |
| B004 | 숲 BGM이 중복 재생  | 숲에 재입장            | 낮음  | 완료   |

심각도 기준은 이렇게 잡습니다.

```text
치명적: 게임 진행 불가, 저장 손상, 실행 불가
높음: 주요 기능 오류, 퀘스트 진행 불가
중간: 불편하지만 우회 가능
낮음: 시각적 문제, 오탈자, 작은 연출 문제
```

출시 전에는 적어도 **치명적 버그와 높은 심각도 버그는 모두 해결**해야 합니다.

---

# 31단계. 밸런스와 난이도를 조정한다

어드벤처 게임의 난이도는 전투 수치보다 **힌트의 양, 동선, 퍼즐 복잡도**에서 결정됩니다.

난이도를 낮추는 방법:

```text
목표 위치를 더 명확하게 보여준다.
NPC가 힌트를 한 번 더 말하게 한다.
중요한 아이템을 더 눈에 띄게 만든다.
잘못된 선택에도 피드백을 준다.
퍼즐 실패 시 다시 쉽게 시도하게 한다.
```

난이도를 높이는 방법:

```text
힌트를 간접적으로 준다.
아이템 위치를 살짝 숨긴다.
퍼즐 단계를 2단계로 늘린다.
이전 대화 내용을 기억해야 풀 수 있게 한다.
```

초반 게임은 쉬워야 합니다.
플레이어가 조작과 규칙을 배우기 전에 어려운 퍼즐을 주면 이탈합니다.

# 32단계. 최적화를 한다

2D 게임도 최적화가 필요합니다. 특히 모바일 웹이라면 더 중요합니다.

확인할 것:

```text
프레임이 안정적인가?
이미지 용량이 너무 크지 않은가?
사용하지 않는 에셋을 로드하고 있지 않은가?
BGM이 중복 재생되지 않는가?
맵이 너무 큰 하나의 이미지로 되어 있지 않은가?
애니메이션이 너무 많은 오브젝트에서 동시에 재생되지 않는가?
모바일 브라우저에서 터치 지연이 없는가?
```

웹게임에서는 특히 다음을 신경 써야 합니다.

```text
초기 로딩 시간
이미지 압축
오디오 로딩
화면 비율 대응
터치 입력
브라우저별 호환성
저장 데이터 localStorage 또는 서버 저장
```

# 33단계. 완성 기준을 정한다

게임은 끝없이 수정할 수 있습니다.
그래서 “완성 기준”을 미리 정해야 합니다.

예시:

```text
처음부터 엔딩까지 플레이 가능하다.
진행 불가 버그가 없다.
모든 대화가 정상 출력된다.
모든 아이템을 정상 획득할 수 있다.
모든 퀘스트가 완료 가능하다.
저장/불러오기가 정상 작동한다.
모바일 화면에서 조작 가능하다.
BGM과 효과음이 정상 재생된다.
엔딩 후 다시 시작할 수 있다.
```

이 기준을 통과하면 출시 가능한 상태로 봅니다.

# 35단계. 출시 후 피드백을 반영한다

# 처음 만들 게임의 추천 사양

처음 개발하는 2D 어드벤처 게임은 이 정도가 적당합니다.

```text
플레이 시간: 10~20분
맵: 2~3개
NPC: 3~5명
아이템: 5~8개
퀘스트: 2~3개
퍼즐: 2개

```

전투, 상점, 장비, 랜덤 생성, 온라인 기능은 두 번째 게임부터 넣는 것이 좋습니다.

# 가장 좋은 첫 번째 목표

처음에는 아래 미니 게임을 완성해보는 것을 추천합니다.

> **“잃어버린 열쇠를 찾아 문을 열고 숲에 들어가는 5분짜리 2D 어드벤처”**

구성:

```text
마을 맵 1개
플레이어 1명
NPC 1명
상자 1개
열쇠 1개
문 1개
대화 5줄
엔딩 메시지 1개
```

이걸 완성하면 2D 어드벤처 게임의 핵심 구조를 거의 다 경험하게 됩니다.

그다음에 다음 기능을 하나씩 추가하면 됩니다.

```text
두 번째 NPC
두 번째 맵
인벤토리 UI
퀘스트 목록
음악 퍼즐
저장 기능
엔딩 연출
```

결론적으로 2D 어드벤처 게임 개발의 핵심은 **큰 세계를 상상하는 것**이 아니라, 먼저 **작은 세계를 끝까지 작동하게 만드는 것**입니다.
처음부터 끝까지 플레이 가능한 5분짜리 게임을 만든 뒤, 그 구조를 10분, 20분, 1시간짜리 게임으로 확장하는 방식이 가장 안전합니다.
